## エグゼクティブサマリー

1. **LLMトレーディングエージェントは「条件付きで」市場に勝てる** — TradingAgentsは3ヶ月テストでSharpe 5.6、+23%リターンを達成。ただしStockBenchベンチマーク（2025年3-6月）では大半のLLMエージェントがBuy&Holdに負けた。
2. **マルチエージェント構成が単一エージェントを大幅に上回る** — ファンダ+センチメント+テクニカル+リスク管理の分業がキー。
3. **Claude 4.6は主要フレームワーク全てで対応済み** — TradingAgents v0.2.2, claude-trading-skills, Claude API直接統合の3経路。
4. **コスト最適化で月$20以下で運用可能** — Haiku（日常監視）+ Sonnet（分析）+ Opus（重大判断）のルーティング。
5. **最大のリスクは過信** — LLMは下落相場で弱い。従来テクニカル戦略との併用が最も堅実。

## 1. 三つのアプローチの比較

### 1-1. アプローチ別メリット・デメリット

| 項目 | claude-trading-skills | TradingAgents | Claude API直接統合 |
|------|----------------------|---------------|-------------------|
| **導入難易度** | ★☆☆ 最も簡単 | ★★☆ 中程度 | ★★★ 実装必要 |
| **カスタマイズ性** | 低（スキル単位） | 中（エージェント設定） | 高（完全制御） |
| **リアルタイム性** | 低（手動実行） | 中（バッチ的） | 高（ループ内統合） |
| **コスト** | Claude Code利用料内 | API費用（月$50-200） | API費用（月$10-50） |
| **実運用実績** | 分析支援のみ | 研究ベンチマーク | 個人実験あり |
| **本システム統合** | 低（分離） | 中（別プロセス） | **高（TradingLoop内）** |

### 1-2. 各アプローチの詳細

**claude-trading-skills**
- セクターローテーション、Distribution Days検出、テクニカルチャート
- Claude Code のスキルとして追加するだけ
- 分析結果は人間が判断 → 手動売買
- **メリット**: ゼロコスト、すぐ使える、学習ツールとして優秀
- **デメリット**: 自動取引不可、リアルタイム不可

**TradingAgents (v0.2.2)**
- 5役割マルチエージェント: ファンダ/センチメント/テクニカル/トレーダー/リスク管理
- Claude 4.6 対応済み（ANTHROPIC_API_KEY設定のみ）
- LangGraph ベースで DAG 実行
- **メリット**: 研究論文付きの実績（Sharpe 5.6）、設計が洗練
- **デメリット**: バックテスト向け、リアルタイム取引は自前実装、API費用高め

**Claude API 直接統合**
- 本システムの TradingLoop 内で Claude を呼び出し
- tool_use でブローカーAPI操作可能
- ハーネスレイヤーでリスク制御
- **メリット**: 最大の柔軟性、既存OMS/DD制御と統合、コスト制御可能
- **デメリット**: 実装コストが最も高い

## 3. マルチエージェント構成の優位性

### 3-1. 単一エージェント vs マルチエージェント

| 指標 | 単一LLM | マルチエージェント | 差分 |
|------|---------|-----------------|------|
| 累積リターン | +17.1% | **+23.2%** | +6.1pp |
| Sharpe Ratio | 3.53 | **5.60** | +2.07 |
| MaxDD | -8.2% | **-5.1%** | 3.1pp改善 |

*(TradingAgents論文、AAPL/GOOGL/AMZN、2024Q1)*

### 3-2. なぜマルチエージェントが強いか

1. **情報の多様性**: テキスト（ニュース/決算）+ 数値（テクニカル）+ 心理（センチメント）を統合
2. **専門特化**: 各エージェントが1つの分析に集中 → プロンプトが短く精度向上
3. **牽制効果**: リスク管理エージェントがトレーダーの暴走を抑制
4. **人間のトレーディングデスクを模倣**: 実世界で機能する分業構造

### 3-3. 本システムへの適用

| 役割 | 実装先 | モデル | コスト/回 |
|------|--------|--------|----------|
| テクニカル分析 | 既存戦略 (Bollinger/MA) | CPU (LLM不要) | $0 |
| センチメント分析 | Claude API (新規) | Haiku | ~$0.002 |
| ファンダメンタル分析 | Claude API (新規) | Sonnet | ~$0.01 |
| 総合判断 | Claude API (新規) | Sonnet | ~$0.01 |
| リスク管理 | 既存 (DD/CB/VaR) | CPU (LLM不要) | $0 |
| 重大判断 (DD>5%時) | Claude API | Opus | ~$0.03 |

**1日のコスト見積**: 24回呼び出し × $0.02 ≈ **$0.48/日 ≈ $15/月**

## 4. Claude API tool_use 統合パターン

### 4-1. 推奨アーキテクチャ

```
市場データ API
    │
    ▼
Data Fetcher (Python)
    │
    ▼
Context Builder (OHLCV + ニュース + センチメント)
    │
    ▼
Claude API (messages.create + tools)
    │
    ▼
Structured Decision (BUY/SELL/FLAT + confidence + reasoning)
    │
    ▼
Harness Validator (ハードルール、リスクチェック)
    │
    ├─ [APPROVED] → OMS → ブローカー
    └─ [REJECTED] → ログ + スキップ
```

### 4-2. ハーネスレイヤーが不可欠な理由

> 「LLMは幻覚（ハルシネーション）を起こす。高リスク判断を100%任せてはならない。」

ハーネスがLLM判断を**上書き**するケース:
- DD > 8% → LLMがBUYと言っても**拒否**
- サーキットブレーカーOPEN → 全判断を**拒否**
- 1日の損失 > 3% → 残りの取引を**停止**
- 同一銘柄に連続3回BUY → 過剰集中を**制限**

### 4-3. Claudeに渡すコンテキスト設計

```python
context = f"""
## 市場状況 ({datetime.now().isoformat()})

### {symbol} テクニカル指標
- 終値: {close:.2f} (前日比: {change_pct:+.2f}%)
- RSI(14): {rsi:.1f}
- MACD: {macd:.4f} (Signal: {signal:.4f})
- BB位置: {bb_position:.2f} (0=下限, 1=上限)
- 出来高: {volume:,.0f} (20日平均比: {vol_ratio:.1f}x)

### 直近ニュース (最新3件)
{news_summary}

### ポートフォリオ状態
- 残高: {balance:,.0f}円
- 現在DD: {current_dd:.1f}%
- エクスポージャー: {exposure:.0f}%
- 既存ポジション: {positions}

### テクニカル戦略のシグナル
- Bollinger RSI ADX: {bollinger_signal}
- MA Crossover: {ma_signal}

## 指示
上記を分析し、以下のJSON形式で回答してください:
{{"action": "BUY"|"SELL"|"FLAT", "confidence": 0.0-1.0, "reasoning": "..."}}
"""
```

## 5. LLM vs 従来手法 — 過学習・ドリフト対応

### 5-1. LLMの優位点

| 能力 | 従来クオンツ | LLM | 優位性 |
|------|------------|-----|--------|
| ニュース解釈 | 困難（NLP前処理必要） | **ネイティブ対応** | LLM |
| 決算書分析 | ルールベース | **文脈理解** | LLM |
| レジーム判定 | HMM/統計モデル | **自然言語で状況説明** | LLM |
| ブラックスワン | 過去データに依存 | **類推能力** | LLM |
| バックテスト | **厳密・再現可能** | 非決定的 | 従来 |
| 低レイテンシ | **μs〜ms** | 100ms〜数秒 | 従来 |
| コスト | **ほぼゼロ** | API費用発生 | 従来 |

### 5-2. ベストプラクティス: ハイブリッド

> **テクニカル戦略（高速・低コスト・再現可能）でシグナル生成 →
> LLM（ニュース・センチメント・文脈理解）で確認・拒否権**

このパターンにより:
- テクニカルが「BUY」→ LLMがニュースで「決算ミス発覚」を検出 → **FLAT** (損失回避)
- テクニカルが「FLAT」→ LLMが「セクター全体の上昇トレンド」を検出 → **BUY** (機会獲得)

### 5-3. ドリフト対応でのLLM活用

既存の DriftDetector (ADWIN/統計) は**数値的な**ドリフトしか検出できない。
LLMを追加することで:
- 「FRBが利上げ転換」→ **レジーム変化を言語から直接検出**
- 「地政学リスク急上昇」→ **テクニカル指標に先行して警告**
- 「決算シーズン開始」→ **ボラ上昇を事前予測**

## 7. 2025-2026年の最新研究

### 7-1. 主要論文

| 論文 | 発表 | 結論 |
|------|------|------|
| **TradingAgents** (arXiv:2412.20138) | 2024/12 | マルチエージェントが単一を+6.1pp超過。Sharpe 5.6 |
| **StockBench** (arXiv:2510.02209) | 2025/10 | 大半のLLMはBuy&Holdに負ける。下落相場が弱点 |
| **FinMem** (ICLR Workshop) | 2024 | 階層メモリで長期文脈を保持。IJCAI 2024コンペ優勝 |
| **Agent Market Arena** (arXiv:2510.11695) | 2025/10 | リアルタイムLLMトレード初の統一ベンチマーク |
| **Can LLM Outperform Long Run?** (arXiv:2505.07078) | 2025/5 | 長期では従来手法に収束する傾向 |

### 7-2. 研究からの教訓

1. **短期（3ヶ月）ではLLMが圧勝することがある** — ただし生存者バイアスに注意
2. **長期（1年以上）では差が縮まる** — 市場効率性が働く
3. **下落相場での損失管理がLLM最大の弱点** — DrawdownController必須
4. **マルチモーダル入力（テキスト+数値）が鍵** — テクニカルだけでは不十分
5. **ハーネス（リスク制御）がないLLMトレーダーは危険** — 幻覚→大損失

## Key Takeaways

### 本システム (AI Trader v2.0) への推奨

1. **Claude API 直接統合を最優先で実装** — TradingLoop内でSonnet呼び出し、ハーネスで安全制御
2. **claude-trading-skills を分析ツールとして併用** — 手動分析・戦略開発に活用
3. **TradingAgents は参考設計として** — マルチエージェント構成のアイデアを取り込む
4. **テクニカル戦略を捨てない** — LLMは「拒否権」として使い、ベースはBollinger/MAを維持
5. **コスト制御: Haiku中心、Sonnet確認用** — 月$15以下で運用可能
6. **DrawdownController/CircuitBreakerがLLMを上書き** — 安全弁は非LLM

### 統合による具体的メリット

| メリット | 根拠 | 期待効果 |
|---------|------|---------|
| **ニュース暴落回避** | LLMが決算ミス/地政学を読む | DD 3-5pp改善 |
| **センチメント反転検出** | SNS/ニュースの感情分析 | エントリー精度向上 |
| **レジーム適応** | LLMが「利上げ転換」等を言語検出 | 従来DriftDetectorの補完 |
| **取引根拠の言語化** | reasoningフィールド | 振り返り・改善に活用 |
| **マルチアセット統合判断** | 株式/暗号/債券の相関を総合 | ポートフォリオ精度向上 |
